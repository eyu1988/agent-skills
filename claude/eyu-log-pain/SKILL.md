# log-pain

## 路径常量

- 登记本：`{{PAIN_DIR}}/01-痛点登记本.md`
- 解决方案库：`{{PAIN_DIR}}/02-解决方案库.md`
- 痛点记录目录：`{{PAIN_DIR}}/痛点记录/`

## 执行步骤

### 第一步：解析痛点

- 如果用户未提供内容，先询问用户描述痛点，等待回复后再继续
- 根据描述自动判断：
  - 域：`WORK`（工作效能）/ `TECH`（研发决策）/ `INFL`（技术影响力）/ `GROW`（个人成长）/ `FINC`（理财投资）/ `LIFE`（生活健康）/ `KIDS`（育儿教育）
  - 严重程度：🔴 阻塞型（影响正常推进）/ 🟡 消耗型（持续消耗精力）/ 🟢 优化型（可以更好但不紧急）

### 第二步：生成 ID

- Read `{{PAIN_DIR}}/01-痛点登记本.md`
- 统计今日（当前日期）该域已有的条目数，序号 +1
- 格式：`域代码-YYYYMMDD-序号`（如 `FINC-20260516-002`）

### 第三步：5Why 根因分析

对痛点做简洁的 5Why 分析，找到根本原因（不超过 5 行）。

### 第四步：写入登记本

将以下内容追加到 `{{PAIN_DIR}}/01-痛点登记本.md` 对应严重程度分区：

```
#### {ID} · {痛点标题} · {日期}

| 字段 | 内容 |
|------|------|
| **域** | {域代码} |
| **严重程度** | {严重程度} |
| **状态** | 🆕 新增 |
| **登记日期** | {日期} |

**痛点描述**：{用户原始描述的结构化版本}

**根因（5Why）**：
{5Why分析}

**解决方案**：→ 待分析
```

同时更新文末统计概览表中对应域的"活跃"计数 +1。

### 第五步：询问用户

告知已登记成功（显示 ID），然后问：

> 是否现在深入分析这个痛点，给出解决方案？（是/否）

- 否 → 结束，提示用户随时说"分析痛点 {ID}"来继续
- 是 → 执行第六步

### 第六步：深入分析并创建三层文件

每个痛点**始终**创建自己专属的三层文件，不与同域其他痛点共享、不追加进已有文件：

1. `{域}-①{时机}-{方法论名称}.md` — 解决方案核心规则、决策框架、操作清单
2. `{域}-②{时机}-{记录名称}.md` — 每次执行时的记录模板（日志、打卡、操作记录）
3. `{域}-③{周期}做-{复盘名称}.md` — 周期性回顾模板（进度、卡点、效果验证）

时机词按域选择：

| 域 | ① 时机 | ② 时机 | ③ 周期 |
|----|--------|--------|--------|
| WORK / TECH | 决策前查 | 执行时填 | 每周做 |
| GROW | 学前查 | 学后填 | 每月做 |
| FINC | 买前查 | 下单时填 | 每周日做 |
| LIFE / KIDS | 用前查 | 执行时填 | 每月做 |
| INFL | 写前查 | 发布时填 | 每月做 |

文件存放路径：`{{PAIN_DIR}}/痛点记录/{域}-{域序号}-{痛点短标题}/`

（文件夹名 = 域代码 + 该域序号 + 两到四字短标题，如 `WORK-001-计划执行落地`）

**完成后**：

- 更新登记本中该条目的"解决方案"字段，链接到 ① 规则层文件
- 在 `{{PAIN_DIR}}/02-解决方案库.md` 对应域下追加方案摘要（问题模式、根因、解决路径、效果验证、对外输出潜力）
