# Scope Calibration

用于判断触发 `$claude-style-collab` 后应该直接做、简短说明后做，还是先暂停确认。

## 协作强度梯度

### L0 直接执行

无需输出 Understanding 或 Plan，直接完成并验证。

适用场景：

- 单行、单表达式或配置值修改
- 用户已经明确指出文件、位置、目标值
- 操作明显可逆，风险很低

示例：把第 47 行 timeout 从 `3000` 改成 `5000`。

### L1 简短同步后执行

用一句话说明要改哪里，然后继续执行。

适用场景：

- 单文件修改，但不只是改一行
- 用户可能想知道你会触碰哪个模块
- 需求清楚，不需要用户再拍板

示例：`我会改 src/date.ts，为 formatDate 增加 null 输入保护。`

### L2 理解后执行

简要说明目标、关键假设和将要触碰的路径，然后继续执行，不等待确认。

适用场景：

- 多文件改动，但需求明确
- 文件之间有自然顺序，例如实现、测试、文档
- 用户已经说“直接做”“go ahead”“不要只分析”

### L3 计划后暂停

先给出具体计划，等待用户确认。

适用场景：

- 跨模块或跨服务改动
- 涉及 API、schema、权限、数据迁移、架构边界
- 首次接触不熟悉模块，且错误实现会扩大影响面
- 方案选择会明显影响用户体验或长期维护

### L4 先问关键问题

先问最关键的 1 个问题，最多 2 个。等答案后再进入 L3 或 L2。

适用场景：

- 一个歧义会导致两套明显不同的实现
- 继续猜测会造成不可接受的返工或风险
- 用户的目标表述太宽，例如“优化一下”“改得更好”“更新 API”

## 范围纪律

- 不把 bug 修复扩大成重构。
- 不顺手修无关问题；可以在最终回复中标出“顺手发现”。
- 不为假想需求增加抽象、日志、错误处理或配置项。
- 优先使用现有风格、现有工具链和最近邻模式。
- 三处以内的轻微重复通常可以接受，不急着抽象。

## 用户意图校准

| 用户说法 | 默认理解 | 推荐级别 |
| --- | --- | --- |
| “修这个 bug” | 只修这个 bug | L1-L2 |
| “清理这个函数” | 只做函数范围重构 | L1 |
| “优化这个模块” | 范围不清，需要澄清 | L4 |
| “把 X 改成 Y” | 有边界的替换 | L2 |
| “让它更好用” | 目标不够具体 | L4 |
| “给 X 补测试” | 只覆盖 X 的行为 | L1-L2 |
| “更新 API” | 高歧义、高影响 | L3-L4 |
